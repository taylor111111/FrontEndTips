

# Frontend Problem Space  
> 用工程视角，而不是 API 视角，理解前端问题

---

## 为什么需要「前端问题空间」

在实际工作和面试中，前端问题经常被混在一起讨论：

- 有的问题在问 **API 会不会**
- 有的问题在问 **库熟不熟**
- 有的问题实际上是在问 **工程能力**

如果不区分这些问题的层级，很容易出现两种情况：
- 工程问题被降级成“记忆题”
- API 细节被错误地当成能力上限

**本文的目标**：  
给前端问题一个清晰的「问题空间划分」，帮助判断：
- 这是什么层级的问题
- 应该用什么方式去回答或解决
- 哪些问题值得投入精力，哪些不值得

---

## 前端问题的三类划分

从工程视角看，前端问题可以分为三类：

1. **工程级问题（Engineering-level Problems）**
2. **API 使用风险问题（API Risk-level Problems）**
3. **库与方案理解问题（Solution / Library-level Problems）**

这三类不是并列的技能点，而是**不同抽象层级的问题空间**。

---

## 一、工程级问题（Engineering-level）

### 定义

工程级问题关注的是：

- **问题是否真实存在**
- **问题的边界是什么**
- **是否值得解决**
- **解决方案的 trade-off 是什么**

它们不是在问“怎么写代码”，而是在问：

> **“我们到底在解决什么问题？”**

---

### 常见工程级问题示例

- 是否需要 **BFF** 来治理前后端边界？
- 项目规模到什么程度，**Monorepo** 才有意义？
- 是否应该引入 **通用组件体系**？
- 复杂度来自哪里？是业务、协作，还是技术选择？

这些问题的特点是：

- 没有标准答案
- 强依赖上下文
- 错误代价高
- 必须显式讨论 **non-goal**

---

### 工程级问题的判断信号

一个人在讨论工程级问题时，通常会提到：

- 问题背景与约束
- 为什么不选择其他方案
- 哪些问题 **明确不解决**
- 未来规模变化下的风险点

如果一个讨论只剩下“怎么实现”，而没有“是否该做”，  
那工程问题已经被错误降级了。

---

## 二、API 使用风险问题（API Risk-level）

### 定义

这一类问题不关心：

> “你知道多少 API”

而关心：

> **“哪些 API 容易被用错，用错的代价有多高？”**

这是**工程风险管理能力**，而不是记忆能力。

---

### 典型高风险 API 示例

- `Array.sort()` —— 隐式 mutation
- `useEffect` deps —— 副作用边界
- `this` —— 不稳定绑定
- `Date` —— 隐含时间依赖
- Promise 并发组合 —— 失败策略与竞态

这些 API 的共同点：
- 本身不复杂
- **容易被误用**
- 出错后影响范围大

---

### 正确姿势

对于 API 风险问题，正确目标是：

- 知道**什么时候不该用**
- 知道**如何约束使用方式**
- 知道**什么时候应该查文档**

而不是：
- 背诵所有行为细节
- 即时回答低频语义

---

## 三、库与方案理解问题（Solution / Library-level）

### 定义

在工程视角下：

> **库 = 对一类工程问题的解决方案**

因此，“熟悉某个库”不等于：
- API 用得多
- 细节背得熟

而等于：
- 知道它解决了什么问题
- 适用边界在哪里
- 常见误用是什么
- 不用它会发生什么

---

### 示例

- **React Query**  
  → 解决异步状态一致性、缓存、并发与失效问题

- **Redux / Zustand**  
  → 解决跨组件状态一致性与可预测性问题

- **组件库（如 Antd）**  
  → 解决 UI 规范收敛与协作成本问题

讨论库时，真正有价值的问题是：
- 为什么需要它？
- 哪些问题它**不解决**？
- 在什么规模下会成为负担？

---

## 为什么要区分这三类问题

如果不区分问题空间，会导致：

- 工程问题被 API 题替代
- 库理解被降级为“会不会用”
- 复杂度来源被掩盖

清晰的分类可以帮助：

- 面试中判断问题是否合理
- 工作中避免错误投入
- 团队中建立共同的工程语言

---

## 一个简单的自检问题

当你遇到一个前端问题时，可以问自己：

> 这是一个  
> - 工程级问题？  
> - API 使用风险问题？  
> - 还是方案/库选择问题？

如果问题无法归类，  
通常意味着**问题本身还没有被正确建模**。

---

## 结语

前端不是 API 的集合，也不是库的堆叠。

前端工程的核心能力在于：

> **识别问题 → 划定边界 → 选择方案 → 控制风险**

理解「前端问题空间」，  
是迈向工程成熟度的重要一步。
