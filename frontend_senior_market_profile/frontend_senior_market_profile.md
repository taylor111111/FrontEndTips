

# Frontend Senior Engineer – Market Profile

## 一、现实市场里的「前端高级工程师」画像（真实，不好听）

这一节不是理想描述，而是**招聘市场真实在筛选什么样的人**。

一句话总结：

> **前端高级工程师 = 熟练执行者 + 稳定交付者 + 低风险个体**

而不是：
- 架构设计者
- 工程问题定义者
- 技术方向探索者

---

### 1. 市场最看重的能力（排序非常重要）

#### 1️⃣ 熟练 API（硬门槛）

包括但不限于：
- JavaScript 基础 API（Array / Object / Promise / Date）
- 浏览器 API（DOM、事件、fetch）
- React 核心 API（useState / useEffect / useMemo / useCallback）
- 常用第三方库 API（antd / axios / react-query 等）

**评价标准不是“你理解得多深”，而是：**
- 会不会用
- 会不会用错
- 出问题能不能快速查文档解决

---

#### 2️⃣ 稳定交付业务功能

市场对“高级”的核心期待是：

> **给你一个中等复杂度的业务需求，你能不能按时、稳定、不出事故地交付。**

典型场景：
- 列表 + 查询条件
- 表单提交与校验
- 弹窗 / 抽屉
- 权限控制 + 路由联动
- 接口联调与状态处理

**重点不是优雅，而是稳定。**

---

#### 3️⃣ 低事故率（非常重要）

相比“能力上限”，市场更在意：

- 不在 render 中 setState
- 不写有副作用的 render 逻辑
- useEffect 依赖不乱写
- 不随意修改引用数据
- 不制造难以 debug 的异步逻辑

一句话概括：

> **高级 = 不给团队制造额外不确定性**

---

### 2. 市场默认“不要求”的能力

下面这些能力，**在市场评价体系中并不是加分项**，甚至可能是风险源：

- 主动定义工程问题
- 抽象系统级模型
- 推动架构重构
- 主导复杂工程改造
- 为系统性问题兜底

现实判断是：

> **这些事情成本高、风险集中、收益不稳定。**

因此，大多数团队并不鼓励前端高级工程师承担这些职责。

---

### 3. 为什么这是一个“执行型画像”

从组织视角看：

- 工程问题往往属于：
  - 架构层
  - 平台层
  - 极少数 owner
- 前端高级工程师的主要角色是：
  - **可靠执行**
  - **降低失败概率**
  - **保证业务推进**

所以，这个画像不是“能力不足”，而是**角色定位不同**。

---

### 4. 一个重要结论

如果你发现：

- 自己更擅长问题定义
- 更在意系统性风险
- 对“只用 API 判断问题”感到不适

那不是你不够高级，而是：

> **你的能力模型，与市场主流前端岗位画像不匹配。**

这是一种结构性不适配，而不是个人失败。

---

> 本文档用于帮助你区分：  
> **市场需要的是什么，与你擅长的是什么。**  
> 区分清楚，才能做出不消耗自己的选择。
