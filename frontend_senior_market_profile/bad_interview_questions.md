

# 坏面试问题识别表（Bad Interview Questions）

> 本文不是吐槽面试官，也不是否定基础知识。  
> 它的目标只有一个：**在面试中识别“噪音型问题”，并用最低成本、工程化的方式稳妥回应，防止问题不合理发散。**

---

## 一、什么是「坏面试问题」

这里的“坏”，**不是问题本身错误**，而是指：

- ❌ 问题**脱离工程语境**
- ❌ 问题**直接考记忆细节**
- ❌ 问题**无法映射到真实工程决策**
- ❌ 问题**容易无限追问、不可收敛**

这些问题在现实工程中：
- 很少单独出现
- 不会以“裸问题”形式被解决
- 即使答对，也不能说明工程能力

---

## 二、识别标准（满足任意两条即可判定为噪音）

| 维度 | 判断问题 |
|---|---|
| 工程映射 | 这个问题能否映射到真实工程事故或设计决策？ |
| 风险密度 | 答错会不会在工程中造成**高风险**？ |
| 记忆依赖 | 是否高度依赖死记硬背的细节？ |
| 可查性 | 现实中是否 **写前必查文档**？ |
| 收敛性 | 是否存在明确的“回答终点”？ |

---

## 三、坏面试问题 × 安全回应模板

下面列出**高频但低工程价值**的问题，并给出：

- 为什么不合理
- 30 秒低成本回答
- 防止发散的话术

---

### 1️⃣「你熟悉 ES6 吗？」

**为什么不合理**
- ES6 是语言集合，不是能力点
- 无工程上下文，容易无限展开

**低成本回答**
> ES6 是我日常使用的基础，比如 `let/const`、解构、模块、async/await，这些主要是用来**减少可变状态和副作用**。

**防发散句**
> 在工程里我更关注哪些特性**降低出错率**，而不是特性本身有多少。

---

### 2️⃣「讲一下 Promise / Promise A+ 规范」

**为什么不合理**
- 规范细节 ≠ 工程能力
- 多数工程问题已由库封装解决

**低成本回答**
> 我理解 Promise 的核心价值是**表达异步完成状态**，在工程中主要通过 React Query / 请求库使用，而不是手写复杂 Promise 链。

**防发散句**
> 规范细节我会查文档，工程上我更关心**并发、取消、竞态和错误边界**。

---

### 3️⃣「说一下 event loop / 微任务顺序」

**为什么不合理**
- 脱离真实 bug 场景
- 顺序细节极易遗忘

**低成本回答**
> 我知道 Promise.then 属于微任务，`setTimeout` 是宏任务，这主要影响**状态何时被读取**。

**防发散句**
> 实际工程里我会通过 effect 边界和状态设计，避免依赖事件循环细节。

---

### 4️⃣「讲一下原型链」

**为什么不合理**
- 现代工程中大量被规避
- 阅读成本高，风险大

**低成本回答**
> 我理解原型链是 JS 的继承机制，但工程中我会优先使用组合，避免隐式继承带来的理解成本。

**防发散句**
> 我更关心的是**如何避免原型链成为维护风险**。

---

### 5️⃣「this 的指向规则是什么？」

**为什么不合理**
- `this` 高度依赖调用方式
- 工程中常被显式规避

**低成本回答**
> `this` 是运行时绑定的，我在工程中尽量使用函数参数和闭包，减少对 `this` 的依赖。

**防发散句**
> 与其记规则，不如减少 `this` 出现的场景。

---

### 6️⃣「Promise.all 的行为细节？」

**为什么不合理**
- 不常写
- 写前一定查文档

**低成本回答**
> 我知道 Promise.all 用于并发聚合，但具体 reject 行为我会在使用前确认文档。

**防发散句**
> 这个 API 我更关注**适用场景**，而不是记忆行为细节。

---

## 四、统一回应策略（非常重要）

当你意识到这是噪音问题时，可以统一采用：

> **「工程锚点 → 简要回答 → 收敛句」**

模板：

> 在工程里我更关注的是【工程问题 X】，这个问题更多是【语言 / 规范 / API 层】，我会在需要时查文档确认。

这不是逃避，而是**成熟工程师的边界意识**。

---

## 五、反向判断团队质量（隐藏价值）

如果一个团队：

- 连续追问 API / 规范细节
- 无法回到工程问题
- 不接受“查文档是正常行为”

👉 **大概率不是长期工程友好的环境**

---

## 六、结语

> 高级工程师的能力，不是记住更多细节，  
> 而是**知道什么值得被记住，什么不值得。**

这份表不是为了“怼面试官”，  
而是为了**保护你的时间、精力和工程判断力**。
